## 简介
本文件面向认证系统的开发者与运维人员，系统化阐述基于 JWT 的认证机制实现，覆盖令牌生成、验证与刷新流程；认证控制器接口设计（登录、刷新令牌、登出）；令牌实体与 JWT 管理器、令牌验证器的工作原理；认证 API 的请求参数、响应格式与错误处理；认证中间件的安全策略与会话管理；以及令牌过期处理、并发登录控制与安全最佳实践。同时提供故障排除与性能优化建议。

### 认证服务（AuthService）
- 登录流程：校验用户存在与激活状态、验证密码；获取用户角色与权限；生成访问令牌与刷新令牌；持久化刷新令牌；更新最后登录时间；记录登录日志；返回 TokenResponseDTO。
- 刷新流程：验证刷新令牌有效性；重新生成访问令牌；返回 TokenResponseDTO（不含新刷新令牌）。
- 登出流程：撤销访问令牌（加入黑名单）；清理用户相关缓存（角色、权限、用户信息）。
- 验证流程：委托 TokenValidator 校验访问令牌有效性与黑名单状态。

```mermaid
flowchart TD
Start(["进入 AuthService"]) --> LoginOrRefresh{"登录还是刷新?"}
LoginOrRefresh --> |登录| VerifyCreds["校验用户与密码"]
VerifyCreds --> RolesPerms["获取角色与权限"]
RolesPerms --> GenTokens["生成访问/刷新令牌"]
GenTokens --> SaveRT["保存刷新令牌记录"]
SaveRT --> UpdateLastLogin["更新最后登录时间"]
UpdateLastLogin --> LogSuccess["记录登录日志"]
LogSuccess --> ReturnLogin["返回TokenResponseDTO"]
LoginOrRefresh --> |刷新| ValidateRT["验证刷新令牌"]
ValidateRT --> ReGenAT["生成新的访问令牌"]
ReGenAT --> ReturnRefresh["返回TokenResponseDTO(不含新刷新令牌)"]
LoginOrRefresh --> |登出| RevokeAT["撤销访问令牌"]
RevokeAT --> ClearCache["清理用户缓存"]
ClearCache --> ReturnLogout["返回成功"]
```

### 令牌验证器（TokenValidator）
- 验证访问令牌：检查签名、类型、黑名单、过期。
- 验证刷新令牌：检查签名、类型、黑名单。
- 撤销令牌：将 jti 加入黑名单，存活时间与原令牌剩余有效期一致。
- 黑名单存储：使用 Django 缓存（Redis）存储，键前缀为 token_blacklist:jti。


## 权限系统参数修正

### 权限检查机制的准确性提升
权限系统参数修正显著提升了权限验证的准确性，主要体现在以下方面：

#### HasPermission 权限检查类
- **参数验证增强**：权限代码参数的验证更加严格，确保传入的权限代码格式正确且存在
- **异步权限检查**：通过 `async_has_permission` 方法实现异步权限验证，提高系统响应性能
- **权限缓存优化**：权限检查结果的缓存机制得到改进，减少重复查询数据库的开销

#### HasAnyPermission 权限检查类
- **多权限支持**：支持检查用户是否拥有指定权限列表中的任意一个权限
- **批量权限验证**：能够同时验证多个权限代码，提升权限检查效率
- **权限组合逻辑**：实现 OR 逻辑的权限组合，满足复杂的权限控制需求

#### 权限验证流程优化

## 性能考虑
- 缓存策略：使用 Redis 缓存提升令牌黑名单查询与用户权限/角色缓存命中率；合理设置超时时间。
- 数据库索引：RefreshToken 的 user 与 jti 字段建立索引，加速查询与去重。
- 异步访问：应用服务与仓储均采用异步 ORM 方法，降低阻塞。
- JWT 载荷精简：避免在 payload 中存放大对象，减少签名与传输开销。
- 过期时间与轮换：合理设置 ACCESS/REFRESH 过期时间，开启刷新令牌轮换与黑名单策略，平衡安全性与用户体验。
- 权限缓存优化：权限检查结果的缓存机制得到改进，减少重复查询数据库的开销。

## 故障排除指南
- 登录失败（用户名或密码错误）：检查用户是否存在、是否激活；查看登录日志；确认密码校验逻辑。
- 刷新令牌无效或已过期：确认刷新令牌类型正确、未被撤销；检查令牌剩余有效期与黑名单状态。
- 登出后仍可访问：确认撤销流程已执行（加入黑名单）且缓存已清理；检查客户端是否正确携带 Bearer Token。
- 并发登录冲突：当前实现未强制多处登录互斥；如需并发登录控制，可在登录时清理旧刷新令牌或引入会话维度的 jti 策略。
- 性能问题：检查 Redis 连接与键空间大小；优化数据库索引；评估令牌黑名单命中率。
- 权限验证失败：检查权限代码格式是否正确；确认用户确实拥有相应权限；查看权限缓存状态。

## 附录：认证API接口规范

- 基础信息
  - 基础路径：/api/v1/auth
  - 认证方式：Bearer Token（访问令牌）
  - 内容类型：application/json

- 登录
  - 方法：POST
  - 路径：/login
  - 请求体：UserLoginDTO
    - 字段：
      - username：字符串，必填
      - password：字符串，必填
      - device_info：字符串，可选
  - 响应体：TokenResponseDTO
    - 字段：
      - access_token：字符串，必填
      - refresh_token：字符串，可选
      - token_type：字符串，默认 Bearer
      - expires_in：整数，单位秒，必填
      - user：对象，可选，包含 user_id、username、email
  - 错误：
    - 用户名或密码错误
    - 用户已被停用
    - 其他业务异常

- 刷新令牌
  - 方法：POST
  - 路径：/refresh
  - 请求体：RefreshTokenDTO
    - 字段：
      - refresh_token：字符串，必填
  - 响应体：TokenResponseDTO
    - 字段：
      - access_token：字符串，必填
      - refresh_token：可为 null
      - token_type：字符串，默认 Bearer
      - expires_in：整数，单位秒，必填
      - user：对象，可选
  - 错误：
    - 刷新令牌无效或已过期
    - 刷新令牌类型不正确
    - 刷新令牌已失效

- 登出
  - 方法：POST
  - 路径：/logout
  - 请求头：Authorization: Bearer <access_token>
  - 响应体：MessageResponse
    - 字段：
      - message：字符串，例如"登出成功"
  - 错误：
    - 无特定业务错误，总是返回成功消息

- 通用响应
  - 成功响应：包含 success=true 与 message
  - 错误响应：包含 success=false 与 message，可选 code
